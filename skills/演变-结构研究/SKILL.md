# 演变-结构研究

研究任何对象的根本方法。

## 这是什么

每个东西都有两个根本维度：

1. **演变**（Diachronic / 时间维度）：它怎么走到今天这个样子的
2. **结构**（Synchronic / 当前维度）：它今天处在什么位置、和别的什么发生关系

最深的洞察来自两者的**交汇**——单看任一维度都看不到的判断。

这是认识论的根本二分（赫拉克利特"流变" vs 巴门尼德"存在"、索绪尔历时/共时、社科纵向/横截面、战略管理"行业演变+竞品格局"——所有研究方法的母版）。

**卡兹克的"横纵分析法"是这个方法在"研究陌生领域"场景下的具体封装**。本 skill 比横纵分析更上位、更普适。

## 适用范围

只要对象**值得用"两个根本维度"理解**，就能用：

| 研究对象 | "演变" 是什么 | "结构" 是什么 |
|---------|-------------|-------------|
| 陌生概念/技术 | 学科血统的源头 → 命名时刻 | 同代竞争概念的生态位对比 |
| 产品/工具 | 公司/团队史，关键决策节点 | 同类产品对比，谁在用 |
| 公司/组织 | 创始历程、战略转型 | 行业地位、竞品格局 |
| 人物 | 职业轨迹、关键转折 | 同期同行的对比 |
| 客户 | 怎么走到今天的业务状态 | 当前他在行业里的处境 + 资源结构 |
| 决策 | 这个问题怎么演化到要决策的 | 当前几个选项的对比 |
| 事件 | 远因 + 近因 | 当下各方力量的格局 |
| 情绪/心结 | 这个情绪从哪累积来的 | 当前内部几个动力的拉扯 |

不适用：
- 简单事实查询（"今天天气如何"）
- 单点问题（"这段代码哪里错了"）
- 创作类（"帮我写首诗"）
- 用户已经熟悉的对象（这个方法的价值在"陌生"或"想重新看清"）

## 启动流程

### Step 1：确认对象 + 扫素材

从用户消息识别"研究对象"。模糊就问一句确认。

**自动扫这些目录里和研究对象相关的文件**（用 Glob/grep）：

- `~/Desktop/01-知识体系/08-他山之石/` —— 别人的思想
- `~/Desktop/01-知识体系/01-卡片笔记/` —— 推演中的主题
- `~/Desktop/01-知识体系/07-录音分析/` —— 录音原矿
- `~/Desktop/01-知识体系/02-知识单元/` —— 成型卡片
- `<你的业务目录>` 或 `<你的客户档案目录>` —— 如果对象是客户/项目
- `~/Desktop/02-个人成长/` —— 如果对象是自己的情绪/决策
- `~/Desktop/`（根目录浅扫一层） —— 临时材料

把扫到的文件列给用户，问一句："还有别的素材要补充吗？给路径或直接说没有。"

如果用户提了新路径，加进来。Read 一遍所有素材建立基线。

### Step 2：判断研究对象类型

不同类型对应不同的"演变"和"结构"含义（见上方适用范围表）。

判断后，告诉用户："我把 XX 当作【类型】来研究——演变看 ___，结构看 ___，对吗？"等用户确认或修正。

### Step 3：创建研究文档

立即创建：

- **外部知识/技术/概念** → `~/Desktop/01-知识体系/01-卡片笔记/演变结构_{对象}_{YYMMDD}.md`
- **客户** → `<你的客户档案目录>/{客户名}/演变结构_{YYMMDD}.md`（如目录不存在询问用户）
- **自己的情绪/决策** → `~/Desktop/02-个人成长/演变结构_{主题}_{YYMMDD}.md`
- 不确定 → 问用户

写入框架：

```markdown
# 演变-结构研究：{对象}

> 演变-结构研究法 | {YYYY-MM-DD}
> 研究类型：{陌生概念 / 客户 / 决策 / 情绪 / ...}
> 素材来源：
> - {扫到的素材文件 1}
> - {扫到的素材文件 2}

## 第一段：演变（Diachronic）

（待写）

## 第二段：结构（Synchronic）

（待写）

## 第三段：交汇

（待写）
```

### Step 4：分三段走，每段等反馈

**每写完一段就停下等用户反馈**。绝不一次性输出全部三段。

**这是这个 skill 最重要的纪律**——不是为了慢，是因为：

- 演变最容易"不够深"，必须给用户机会挑刺（实战教训：纵向写完不等反馈直接推进，结果整个研究的根基浅）
- 结构容易遗漏关键对比对象或写成参数表
- 交汇必须基于校准过的前两段才有价值

每段写完后，把内容追加到上面创建的研究文档里（用 Edit 工具），然后在对话里输出该段全文 + 反馈选项。

## 第一段：演变分析（Diachronic）

**目标**：还原研究对象**怎么走到今天**——不是流水账，是看清"它从什么根上长出来的"。

### 关键纪律：必须追到根

**最常见的失败模式**：只追近 3-5 年的近因。这不是演变，是流水账。

**正确做法**：拆解研究对象的核心要素，问自己——

> "这个对象的每个关键特征/词/决策，最早是从哪里来的？追到 1950s 还是 1900s？甚至更早？"

**强制要求**：研究对象诞生 < 50 年的，**必须有至少 1 条线追到对象诞生前 30 年以上的源头**。

### 不同类型对象的"追根"姿势

| 对象类型 | 追根方向 |
|---------|---------|
| 陌生概念/技术 | 学科血统——这个概念里的核心词最早是谁提出的 |
| 产品/工具 | 创始团队的前史 + 同类产品的早期形态 + 它选择的技术栈的源头 |
| 公司/组织 | 创始人的背景 + 当时行业的奠基期 + 战略选择的渊源 |
| 人物 | 职业轨迹 + 师承 + 童年/早期形成期的关键事件 |
| 客户 | 客户所在行业的奠基期 + 客户企业创立背景 + 当前业务模式的演化路径 |
| 决策 | 这个问题第一次出现是什么时候 + 中间错过了什么节点 + 现在为什么必须决策 |
| 事件 | 远因（结构性原因）+ 近因（触发事件）+ 转折时刻 |
| 情绪/心结 | 这个情绪第一次出现在什么场景 + 中间被哪些事件强化 + 现在为什么爆发 |

举例（陌生概念）：

研究 Harness Engineering（2026 命名的概念），不能只追到 2022 ChatGPT。要追到——

- 1948 Wiener 控制论（Feedforward / Feedback 的源头）
- 1950s Deming PDCA + 丰田 TPS（Quality Left 的源头）
- 1986 Brooks Subsumption Architecture（嵌套循环的源头）
- 1990s 软件 Test Harness（这个词本身的源头）

**思考方式**："这个东西里的每个关键要素，都是新的吗？如果不是，原本是干什么用的？"

### 结构

不是写编年史，是写"几条线如何汇成今天的它"：

1. **找 3-5 条源头**：每条独立追溯（年代 + 关键人物/事件 + 它如何流到今天）
2. **关键转折节点**：每个节点的决策逻辑——为什么是这个时机/这群人
3. **汇流时刻**：把几条源头如何在某个标志性事件里拧合
4. **演变价值**：让读者明白看到这些根之后，对象的"根系深度"如何

### 篇幅

弹性，根据对象复杂度调整：

- 简单对象（个人情绪、单个决策）：800-1500 字
- 中等对象（产品、概念、客户）：2000-3000 字
- 复杂对象（公司、行业、思想流派）：3000-5000 字

宁可砍内容也不要灌水。质量 > 篇幅。

### 写作风格

- 叙事驱动，不是罗列。"故事感"串起来
- 每个判断后跟事实支撑（年代/人名/事件名/书名）
- 不确定的明确标注"推测"
- 用人话写，避免咨询术语

### 写完后

追加到研究文档，对话里输出全文，然后问用户：

> 演变走完。下一步？
> 1️⃣ 继续走结构
> 2️⃣ 重写演变（哪里不够深？追到哪里还不够？）
> 3️⃣ 暂停讨论某条源头线
> 4️⃣ 跳到交汇

## 第二段：结构分析（Synchronic）

**目标**：以当前为切面，看研究对象**今天处在什么位置**——和谁发生关系、占什么生态位。

### 第一步：判断"对比对象"是什么

不同研究对象的"结构"对应不同东西：

| 对象类型 | "结构"对比的是什么 |
|---------|-------------------|
| 陌生概念 | 同代竞争概念 / 替代方案 |
| 产品/工具 | 同类竞品 |
| 公司 | 同行业竞争对手 |
| 人物 | 同期同行 |
| 客户 | 同行业其他客户 / 客户内部各业务线 |
| 决策 | 选项 A / B / C 各自的代价与收益 |
| 事件 | 当事各方 / 各方利益结构 |
| 情绪 | 内部几个矛盾动力的拉扯（如：想做 vs 怕被否定）|

### 对比维度（灵活调整）

每个对比对象都要回答：

1. **核心差异**：底层逻辑、目标用户、优势短板
2. **它"活成了什么样"**：派别气质、代表人物、典型场景、外界评价
3. **生态位关系**：和研究对象是替代 / 互补 / 上下位 / 同位
4. **趋势判断**：未来 1-2 年这个对比对象会往哪走

### 关键纪律：要"有温度"

**最常见的失败模式**：写成参数对照表，每个对象功能罗列，没温度。

**正确做法**：每个对比对象 300-500 字小传——讲清楚它的派别气质、谁在用、为什么选它、典型场景、外界怎么评价它。要让读者读完知道这个对比对象"长什么样"，不是知道它"有什么功能"。

举例：写 Evaluation Engineering 不是"专注于评估，代表人物 Hamel Husain"，而是"小而严肃的圈子。Hamel 反复讲'If you're not writing evals, you're not engineering'。企业级团队对这派极度推崇，但在 demo-driven 的创业圈里不算主流（因为写 eval 慢拖进度）。"

### 在最后给一张对比表

不要表格代替分析，但分析完之后用一张表浓缩：核心差异、聚焦侧、代表人物、受欢迎圈子、成熟度、局限。

### 给出生态位结论

明确说出研究对象在这个版图里**占什么位置**——填补空白、正面竞争、上下位包含、还是别的。

### 篇幅

- 简单对象：1000-2000 字
- 中等对象：2500-3500 字
- 复杂对象：3500-5000 字

### 写完后

追加到研究文档，对话里输出全文，然后问用户：

> 结构走完。下一步？
> 1️⃣ 走最后的交汇判断
> 2️⃣ 重写结构某个对比对象
> 3️⃣ 增减对比对象
> 4️⃣ 暂停讨论

---

## 第三段：交汇判断

**目标**：综合判断。**不是前两段的缩写版**，是基于两轴交叉得到的、单独看任何一条都看不到的判断。

这一步对应黑格尔辩证法的"合"——演变（正）+ 结构（反）→ 综合判断（合）。

### 必答 4 个问题

每个问题都必须给出明确判断（不要"可能是 / 也许 / 取决于"这种和稀泥）：

1. **结构性优势的来源**：研究对象今天的核心优势，能追溯到演变里哪个早期节点？（演变 × 结构）
2. **结构性短板的根源**：研究对象今天的核心短板，是不是当年某个合理选择变成的包袱？（演变 × 结构）
3. **真实位置**：结构给出的位置 + 演变积累出的差异化，结合成一个独特定位
4. **未来 1-2 年关键变量**：什么发生了它会变强？什么发生了它会被替代/被颠覆？

### 给一个一句话判断

整篇研究浓缩成一句话——这是用户能记住的唯一一句话。

### 篇幅

500-2000 字（看对象复杂度）。最浓缩的一段，绝不灌水。

### 收尾

追加到研究文档，对话里输出全文，然后问用户三个升级选项：

> 三段全部走完。要不要——
> 1️⃣ 拆出原子卡片到 `02-知识单元/`？（如果产生了独立可读的观点/概念/问题）
> 2️⃣ 发布到飞书 / 公众号？
> 3️⃣ 启动 /溯源 skill 把研究对象追溯成概念卡片？
> 4️⃣ 都不需要，归档收工

## 三个常见失败模式

| 失败模式 | 具体表现 | 强制规则 |
|---------|---------|---------|
| 演变不够深 | 只看近 3-5 年 | 至少 1 条线追到对象诞生前 30 年以上的源头 |
| 结构变参数表 | 每个对比对象罗列功能 | 每个对比对象讲"活成了什么样"——派别+人物+用户+评价 |
| 交汇变缩写 | 把前两段总结一下完事 | 必须新增结构性优势/短板/真实位置/未来变量 4 个判断 |

## 写作总原则

- **可读性优先**：写得像优质深度报道，不是咨询报告
- **叙事驱动**：有起承转合，不是流水账
- **观点要有，但建立在事实之上**：先摆事实再给判断
- **用人话写**：避免咨询套话
- **对比要有温度**：对比对象讲"活成了什么样"，不是参数差异
- **不确定标注**：信息不确定的明确标"推测"，不编造

## 这个方法的根（理解了能灵活变形）

知道你在用什么方法，比熟练用它更重要。这个 skill 的根是——

- **认识论根本二分**：时间维度（赫拉克利特"流变"）vs 结构维度（巴门尼德"存在"）
- **结构主义语言学**：索绪尔的共时/历时分析（1916）—— 提供"两视角必须严格分开"的纪律
- **实证社科**：纵向/横截面研究 —— 提供"两个研究维度"的概念隐喻
- **战略管理**：Porter 五力、SWOT、BCG（1960s-1979）—— 提供"竞品对比"和"生态位"思维
- **黑格尔辩证法**（1807）：正反合 —— 提供"交汇出新判断"的逻辑结构
- **卡兹克横纵分析法**（2024）—— 这个方法在"研究陌生领域"场景下的现代封装

理解这条血统，你能根据具体场景灵活变形——而不是被"横纵"或"演变-结构"这些字眼绑死。
